# 演習4：アーキテクチャ設計

## 1. 目的

演習3で作成したシステム分析モデルをもとに、HRSを実装しやすい設計モデルへ変換する。

具体的には、HRSを複数の層に分け、各層の責務とクラス間の依存関係を明確にする。
また、演習5でプログラムへ落とし込めるように、パッケージ図と設計クラス図を作成する。

## 2. 設計方針

HRSでは、以下の3つのユースケースを中心に設計した。

1. 空室を確認して部屋を予約する
2. 予約番号でチェックインする
3. 宿泊料を精算してチェックアウトする

演習3で識別したバウンダリ、コントロール、エンティティの役割を保ちながら、以下の層に整理した。

| 層 | 役割 | 主なクラス |
|---|---|---|
| ユーザインタフェース層 | 利用者やホテル受付係との入出力を担当する | 予約画面、チェックイン画面、チェックアウト画面 |
| アプリケーション層 | ユースケースごとの処理の流れを制御する | 予約処理、チェックイン処理、チェックアウト処理、予約管理、部屋管理、支払い管理 |
| ドメイン層 | HRSで扱う主要なデータや業務概念を表す | 利用者、ホテル受付係、予約、部屋、宿泊、宿泊料、支払い、空席数 |

## 3. パッケージ図

PlantUMLファイル： [パッケージ図](../diagrams/exercise4_package_diagram.puml)

パッケージ図では、HRSを「ユーザインタフェース」「アプリケーション」「ドメイン」の3つに分けた。

ユーザインタフェース層は、予約画面、チェックイン画面、チェックアウト画面から構成される。
アプリケーション層は、予約、チェックイン、チェックアウトの各処理と、それらを支える管理クラスから構成される。
ドメイン層は、HRSで扱う予約、部屋、宿泊、宿泊料、支払いなどのエンティティから構成される。

## 5. 演習3からの修正点

演習3では、ユースケースを実現するために必要なバウンダリ、コントロール、エンティティを分析した。
演習4では、それらを実装しやすい形に整理し、パッケージ構成とクラス名を具体化した。

特に、予約、チェックイン、チェックアウトの処理を分けたままにし、各処理の責任が混ざらないようにした。
また、予約情報、部屋情報、支払い情報をそれぞれ管理するクラスを用意し、データ操作の責任を分離した。

## 6. 工夫点

ユーザインタフェース層は変更されやすいため、ドメイン層に直接依存しすぎないようにした。
アプリケーション層を間に置くことで、画面をCUIからWeb UIに変更しても、予約、チェックイン、チェックアウトの基本的な処理を保ちやすくした。

また、演習5で実装する際にコードへ落とし込みやすいように、クラス名、属性名、メソッド名をJava風の英語名にした。
